# Contract: Registration API

**Owner**: `stem-dictionaries-manager` (external)
**Consumer**: `Stem.ButtonPanelTester.Infrastructure.Http.HttpRegistrationClient`
**Aligned with**: `stem-dictionaries-manager` v0.7.0 — feature
`001-bootstrap-registration` (`stem-dictionaries-manager/specs/001-bootstrap-registration/contracts/register.md`).
Refinement issue: [stem-dictionaries-manager#54](https://github.com/luca-veronelli-stem/stem-dictionaries-manager/issues/54).

---

## Endpoint

```
POST {Dictionary:BaseUrl}/register
```

Path is `/register`, **not** `/api/register`. The endpoint is
unauthenticated — it predates the `X-Api-Key` middleware and is the
bootstrap entry point.

## Request headers

| Header | Value |
|---|---|
| `Content-Type` | `application/json; charset=utf-8` |
| `Accept` | `application/json` |
| `User-Agent` | `Stem.ButtonPanelTester/<assemblyVersion>` |

No `X-Api-Key` (we don't have one yet — that's why we're calling `/register`).

## Request body

```json
{
  "bootstrapToken": "stbt_<43-char-base64url>",
  "descriptor": {
    "clientApp":   "ButtonPanelTester",
    "osUserId":    "<SHA-256 hex of Windows SID>",
    "machineId":   "<SHA-256 hex of Win32_ComputerSystemProduct.UUID>",
    "installGuid": "<per-installation GUID, non-zero>",
    "appVersion":  "<SemVer 2.0 string>"
  }
}
```

### Field constraints

- `bootstrapToken`: required, non-empty after trim. Validated client-side
  by `BootstrapToken.TryCreate`. Server validates by PBKDF2-hash lookup;
  format is informational.
- `descriptor.clientApp`: required, fixed string `"ButtonPanelTester"`.
  The server's per-`clientApp` policy registry uses this key to look up
  which descriptor fields are mandatory for this consumer (see *Privacy
  posture* + *Per-clientApp policy* below). A lookup miss (e.g. someone
  forks our client and changes the value) lands on the same 401 as
  "token unknown" — see *Failure responses*.
- `descriptor.osUserId`: required for `ButtonPanelTester`. **Hashed**
  per the *Privacy posture* section — raw `WindowsIdentity.GetCurrent().User.Value`
  (the SID) MUST NOT cross the wire.
- `descriptor.machineId`: required for `ButtonPanelTester`. **Hashed**
  per *Privacy posture* — raw `Win32_ComputerSystemProduct.UUID` MUST NOT
  cross the wire.
- `descriptor.installGuid`: required, non-zero `Guid`. Generated by the
  client on first launch and persisted in
  `%LOCALAPPDATA%\Stem\ButtonPanelTester\credentials\install.guid` (text
  file, paired with `credential.dpapi` because the Re-Register flow
  rotates both together) so subsequent launches and re-registrations
  carry the same GUID.
  Zero-Guid is rejected by the server with `InstallGuidInvalid → 400`.
- `descriptor.appVersion`: optional. When sent, MUST be a SemVer 2.0
  string. Source: `AssemblyInformationalVersionAttribute.InformationalVersion`
  on the running `ButtonPanelTester.GUI` assembly. Malformed (e.g.
  `"banana"`, `"v1.0.0"` with the `v` prefix, un-padded `"1.0"`) is
  rejected with `DescriptorMalformed → 400`.

The `bootstrapToken` is supplied by the technician through the
registration dialog. The client passes it through after
`BootstrapToken.TryCreate` validation (trim + non-empty).

## Privacy posture

Reference: `stem-dictionaries-manager/specs/001-bootstrap-registration/contracts/register.md`
§"Privacy posture".

The server treats `osUserId` and `machineId` as opaque strings — it
does not parse, join, or correlate them against any external system.
The server permits raw values for all consumers, **but for
`ButtonPanelTester` (supplier-deployed) the cross-organization data
boundary forces hashing**. STEM must not receive a raw Windows SID or
a raw machine UUID from a supplier's hardware.

Compute both as the **lowercase SHA-256 hex digest of the UTF-8 bytes
of the raw value** (64 characters, `[0-9a-f]`). The hash's
collision resistance (2^256) is sufficient to function as a per-machine
/ per-user fingerprint for revocation and forensics.

This codifies **FR-020** in the consumer-facing form: *do not transmit
raw identifiers; the hashed equivalents are permitted.*

## Successful response

```http
200 OK
Content-Type: application/json

{
  "installationId": 142,
  "apiCredential":  "stak_<43-char-base64url>",
  "issuedAt":       "2026-05-18T10:23:45.000Z"
}
```

The client extracts `apiCredential` and constructs
`InstallationCredential.Create response.apiCredential`. The plaintext
credential is handed to `DpapiCredentialStore.SaveAsync` for
encryption-at-rest. The plaintext is never logged, never written
anywhere except DPAPI-encrypted, never displayed in the UI after the
dialog closes. `installationId` and `issuedAt` are surface area for
richer UX later (e.g. status row tooltip "Installation #142,
registered 2026-05-18") — not consumed in v1.

## Failure responses

Every failure response uses the same envelope: `{ "error": "<short message>" }`.
The **status code** distinguishes the failure class; the body
message is a short developer hint, not a token-validity oracle, and
the consumer does NOT display it to the technician (the dialog
shows the curated user-facing string keyed off `RegistrationError`
below, not the server's `error` field).

### Status → `RegistrationError` map

| HTTP | Server outcome | F# `RegistrationError` | Dialog inline-error message |
|---|---|---|---|
| 200 | `Success` | n/a — `Ok credential` | n/a — dialog closes |
| 400 | `TokenMissing` / `DescriptorMalformed` / `DescriptorMissingField` / `InstallGuidInvalid` | `DescriptorRejected detail` | "The client sent a malformed registration request. Please report this — include the application version and the time of the attempt." |
| 401 | `TokenInvalid` *or* `ClientScopeMismatch` *or* policy-lookup miss | `TokenInvalid` | "The token was not accepted. Check it and try again, or contact STEM for a fresh one." |
| 409 | `TokenAlreadyUsed` | `TokenAlreadyUsed` | "This bootstrap token has already been used. If you need to re-register this machine, contact STEM to revoke the existing installation." |
| 410 | `TokenExpired` | `TokenExpired` | "The bootstrap token has expired. Ask STEM for a fresh one." |
| 423 | `TokenRevoked` | `TokenRevoked` | "The bootstrap token has been revoked. Contact STEM." |
| 5xx | `AuditFailure` (500 specifically) / any other | `RegistrationServerError httpStatus` | "The dictionary service is temporarily unavailable (HTTP <n>). Try again later." |
| network failure | n/a | `RegistrationNetwork NetworkUnreachable` | "Could not reach the dictionary service. Check your network and try again." |
| client timeout (90 s) | n/a | `RegistrationNetwork Timeout` | "The registration request timed out. Try again." |

The dialog renders the message in the `Failed` state. The dialog
stays open so the technician can correct the token and retry (for
401/409/410/423/5xx/network/timeout). For 400 (DescriptorRejected)
the dialog stays open but no correction the technician can make
will help — only a client-side fix or a new build will resolve it.

### Why the 401 conflation

The server folds three causes into a single 401 + identical body:
unknown token, token-scope mismatch, and `clientApp` policy-lookup
miss. This hides which apps a given token was scoped to. The
bootstrap token's entropy (43-char base64url ≈ 2^258) makes
random-guess enumeration infeasible regardless, so the conflation
is defense-in-depth — see `stem-dictionaries-manager` register.md
§"Threat model".

The consumer-side consequence: `TokenInvalid` is genuinely
ambiguous. The dialog message ("token not accepted, contact STEM")
matches that ambiguity honestly — we cannot tell the technician
whether the token was wrong, the install was already done, or our
build is broken.

## Timeout and retries

- Client-side timeout: **90 s** (`HttpRegistrationClient.TimeoutSeconds`), same as the dictionary fetch (uniform user expectation). Raised from the original 10 s in [`../phases/phase-7.md`](../phases/phase-7.md) to absorb Azure App Service Free-tier cold-start latency (issue #92).
- Retries: **none**. The technician is at the keyboard; if it fails
  they retry by clicking Submit again. The dialog's internal Elmish
  loop handles the retry-on-failure UX; the wire client makes no
  retry decisions.

## Idempotency

Not idempotent. A second `POST /register` with the same bootstrap
token returns `409 Conflict` (`TokenAlreadyUsed`). The first response
was the only opportunity to capture the plaintext `apiCredential`.
On network failure mid-call the recovery path is operator-side: admin
revokes any installation that may have been created and mints a new
bootstrap token.
